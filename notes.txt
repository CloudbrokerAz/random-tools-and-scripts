● Here's the flow:

  The pipeline accepts PRs from two sources (line 32-33 of the workflow):

  1. dependabot/terraform/* — PRs created by Dependabot. It scans the private registry
  monthly (configured in .github/dependabot.yml) and opens PRs when new module versions
  are detected.
  2. module-update/* — PRs created by the fallback scanner (scan-module-versions.sh).
  This exists because Dependabot has blind spots — it can miss submodule paths like
  cloudwatch//modules/metric-alarm. The scanner queries the TFC API directly and creates
  PRs in the same format.

  The "Source" field in the step summary is just a label showing which path created the
  PR. The classify-version-bump.sh script checks if the branch name starts with
  dependabot/ — if yes, it's Dependabot; if not, it's "Fallback Scanner / Manual".

  In your demo, trigger-bump.sh creates branches with dependabot/terraform/ prefix, so it
   should show "Dependabot". If you're seeing "Fallback Scanner / Manual", the branch
  name likely didn't match the dependabot/* pattern — maybe it was created with a
  module-update/ prefix or something else.

  The distinction is cosmetic for the pipeline — both sources go through the exact same
  Jobs 1-4. It's just an audit trail so you know how the version bump was detected.
